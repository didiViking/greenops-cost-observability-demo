# 🌱 GreenOps Observability Demo (Azure Billing → SPRUCE → VictoriaMetrics → Grafana)

![License](https://img.shields.io/badge/license-MIT-blue.svg)
![Status](https://img.shields.io/badge/status-demo-orange.svg)
![Stack](https://img.shields.io/badge/stack-Azure%20%7C%20SPRUCE%20%7C%20VictoriaMetrics%20%7C%20Grafana-green.svg)

This repository demonstrates an end-to-end **GreenOps observability pipeline** using **Azure billing exports**, **SPRUCE**, **VictoriaMetrics**, and **Grafana**.

Instead of treating cloud costs as static billing reports, this project transforms them into **Prometheus-compatible time series metrics** that can be queried, aggregated, visualized, and correlated alongside infrastructure telemetry.

The repository accompanies my GreenOps observability blog series.

---

# Why This Demo?

Traditional FinOps workflows typically stop after generating reports or dashboards.

This project demonstrates a different approach:

> **Cloud cost data is telemetry.**

Once billing information becomes metrics, we can:

- query it with PromQL
- visualize it alongside infrastructure metrics
- build alerts
- detect anomalies
- correlate operational events with cloud spending
- expose sustainability indicators through the same observability platform

---

# What You'll Build

By following this repository you'll build an end-to-end GreenOps observability pipeline capable of:

- importing Azure billing exports
- normalizing billing data using SPRUCE
- converting billing records into Prometheus metrics
- storing them in VictoriaMetrics
- exploring cost and sustainability metrics with VMUI
- visualizing them with Grafana dashboards

# Why SPRUCE?

This project intentionally separates **cost normalization** from **observability**.

SPRUCE is responsible for:

- normalizing Azure billing exports
- producing standardized FOCUS datasets
- enriching billing metadata
- exporting Parquet datasets

SPRUCE **does not generate observability metrics**.

Learn more:

- https://opensourcegreenops.cloud/latest/
- https://github.com/digitalpebble/spruce

---

# Why VictoriaMetrics?

Once billing data becomes metrics, we gain everything an observability platform already provides:

- PromQL querying
- dashboards
- aggregation
- alerting
- historical analysis
- label filtering
- cardinality analysis

Instead of treating cloud costs as spreadsheets, they become first-class operational telemetry.

Learn more:

- https://github.com/victoriametrics/VictoriaMetrics
- https://docs.victoriametrics.com/

# Getting Started

## 1. Export Azure billing data

This repository supports both:

- Azure Legacy Cost Export
- Azure FOCUS Export

Both can be converted into a format accepted by SPRUCE.

---

## 2. Convert Azure export into SPRUCE input

Legacy export:

```bash
python azure_legacy_to_spruce_csv.py
```

FOCUS export:

```bash
python focus_to_spruce_csv.py
```

Both produce:

```
outputs/spruce_input.csv
```

---

## 3. Run SPRUCE

```bash
docker run --rm \
  -v "$(pwd)/outputs:/outputs" \
  ghcr.io/digitalpebble/spruce:latest \
  -p AZURE \
  -i /outputs/spruce_input.csv \
  -c /outputs/config.json \
  -o /outputs/result
```

SPRUCE generates a normalized Parquet dataset:

```
outputs/result/
    part-xxxxxxxx.snappy.parquet
```

---

## 4. Start VictoriaMetrics

```bash
docker run -d \
  --name victoriametrics \
  -p 8428:8428 \
  victoriametrics/victoria-metrics
```

VMUI:

```
http://localhost:8428/vmui
```

---

## 5. Export GreenOps metrics

```bash
python spruce_to_victoriametrics.py
```

The exporter reads the Parquet dataset generated by SPRUCE and converts every billing record into Prometheus metrics before importing them into VictoriaMetrics.

---

## 6. Start Grafana

```bash
docker run -d \
  --name grafana \
  -p 3000:3000 \
  grafana/grafana
```

Login:

```
admin
admin
```

Configure a Prometheus datasource pointing to:

```
http://host.docker.internal:8428
```

(or `http://localhost:8428`, depending on your deployment)

Import:

```
grafana/dashboard.json
```

## Generated Metrics

The exporter produces three Prometheus metrics:

### cloud_cost

Represents the monetary cost of each billing record.

Labels include:

- provider
- service
- category
- region
- resource_group
- subscription
- sku

Example:

```promql
sum(cloud_cost)
```

```promql
sum by(service)(cloud_cost)
```

```promql
topk(10, sum by(sku)(cloud_cost))
```

---

### cloud_co2_estimate

Estimated CO₂ emissions derived from the cloud cost using a simple regional emission factor.

Labels:

- provider
- service
- category
- region
- resource_group
- subscription

Example:

```promql
sum(cloud_co2_estimate)

sum by(region)(cloud_co2_estimate)
```

---

### cloud_resource_info

Inventory metric describing cloud resources.

Value is always `1`, allowing PromQL aggregation with `count()`.

Labels include:

- provider
- service
- category
- region
- resource_group
- subscription
- sku

Example:

```promql
count by(sku)(cloud_resource_info)

count by(region)(cloud_resource_info)
```

# Disclaimer

This repository is intended for **educational and demonstration purposes**.

It does **not** calculate real carbon emissions.

SPRUCE is used exclusively as a **billing normalization engine**.

The GreenOps metrics, Prometheus metric names, Grafana dashboards, VictoriaMetrics queries, and CO₂ estimation model were designed specifically for this observability demonstration and are **not part of the SPRUCE project**.
